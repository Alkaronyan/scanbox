# SCANBOX — Dynamic Video Switcher Module

A deterministic video source switcher for Raspberry Pi 4. Hot-swaps between a physical USB camera and a synthetic mock source via a REST API and web UI. Accessible over LAN/WiFi or directly via USB cable (no router needed).

## Architecture

### Containerization (strict)
Everything that can run in a container does. The host OS stays minimal. Only Docker Engine, kernel headers, essential host utilities (kmod, procps, udev), configfs mount, git, and systemd service files are installed on the host — all automated by `host/setup_host.sh`. This ensures the stack works even on minimal Debian installations (debootstrap --variant=minbase).

### Containers

| Container | Purpose |
|---|---|
| **Vid_Mux** | GStreamer pipeline, Flask REST API, MJPEG stream, Web UI. Port 80. |
| **Vid_Mux_TEST** | Compiles v4l2loopback v0.15.3 against the host kernel, creates /dev/video200, feeds a live SMPTE test pattern into it. |
| **scanbox_dhcp** | dnsmasq DHCP server on usb0. Assigns IP to the connected Windows PC automatically. |

### USB NCM Direct Link

| Side | IP |
|---|---|
| Pi | 192.168.199.1 |
| Windows PC | 192.168.199.2 (DHCP, /30 point-to-point) |

Windows 11 recognises the Pi as a USB ethernet adapter natively — no drivers to install.

## Quick Start

### Fresh Pi setup (run once)
```bash
sudo ./host/setup_host.sh
sudo reboot
```
This installs Docker, kernel headers, essential host utilities (kmod, procps, udev), configfs mount, systemd services, and generates `.env`. After reboot the full stack starts automatically.

> **Note for minimal distros:** `setup_host.sh` now explicitly installs `kmod` (modprobe/lsmod), `procps` (ps), and `udev` (by-id symlinks) — dependencies that minimal Debian installations may omit. configfs is added to `/etc/fstab` to guarantee the USB gadget setup works.

### Manual start / rebuild after code changes
```bash
./rebuild_vid_mux.sh
```
Detects cameras dynamically, rebuilds the vid_mux image, and relaunches the full stack.

### Access the web UI
| Connection | URL |
|---|---|
| USB cable (direct) | http://192.168.199.1 |
| WiFi / LAN | http://\<pi-ip\> |

## Boot Sequence

```
scanbox-gadget.service  →  USB NCM gadget (usb0 = 192.168.199.1)
scanbox.service         →  rebuild_vid_mux.sh
    ├── scanbox_dhcp     →  DHCP server on usb0
    ├── vid_mux_test     →  compiles v4l2loopback → /dev/video200 (healthcheck)
    │    └── waits for /dev/video200 to appear (polls every 2s, 120s timeout)
    └── vid_mux          →  built with dynamically detected cameras → Flask 
```

`scanbox.service` replaces the old `scanbox-stack.service`. Key advantage: cameras are detected dynamically at every boot, so plugging in a new camera is pick-up-and-go.

## REST API

| Method | Path | Description |
|---|---|---|
| GET | `/` | Web UI |
| GET | `/stream` | Live MJPEG stream |
| GET | `/api/v1/status` | Active source, source list, running source ids |
| POST | `/api/v1/heartbeat` | Keep cameras alive; first call starts all pipelines |
| POST | `/api/v1/source` | Switch source `{"source_id": N}` |
| POST | `/api/v1/source/<id>/start` | Start one source pipeline (debug) |
| POST | `/api/v1/source/<id>/stop` | Stop one source pipeline (debug) |
| POST | `/api/v1/snapshot` | Capture frame. Optional body: `{"filename": "name.jpg"}` |
| GET | `/api/v1/snapshot/last` | Retrieve last snapshot |
| GET | `/api/v1/snapshots` | Paginated snapshot list (`?offset=0&limit=5`) |
| GET | `/api/v1/snapshot/<filename>` | Retrieve a named snapshot |
| DELETE | `/api/v1/snapshot/<filename>` | Delete a named snapshot |
| GET | `/api/v1/camera/controls` | V4L2 controls for the active source (empty for mock) |
| POST | `/api/v1/camera/control` | Set V4L2 control `{"control": "saturation", "value": 128}` |

## Running Tests

The test suite runs entirely inside the `test_runner` Docker container.
The full stack must be running first (`docker compose up -d`).

```bash
# Interactive menu
./run_tests.sh

# Run all 33 tests directly
docker compose --profile test run --rm --no-deps test_runner

# Run a single layer
./run_tests.sh -m layer1    # hardware checks
./run_tests.sh -m layer2    # container health
./run_tests.sh -m layer3    # GStreamer pipeline
./run_tests.sh -m layer4    # API contract
./run_tests.sh -m layer5    # behavioral / visual
```

Tests that pass delete their snapshot files automatically. Tests that fail leave
snapshots in `./snapshots/` for visual inspection.

See [docs/TESTS.md](docs/TESTS.md) for full details on every test and utility.
