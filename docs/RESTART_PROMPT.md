### Project Context & Decisions
1. **Host OS Preservation (Strict Containerization)**: Everything that can run inside a container does. Only Docker Engine, kernel headers, git, systemd services, and essential host utilities (udev, kmod, procps) are installed on the host via host/setup_host.sh. This ensures the stack works on minimal Debian distros (debootstrap --variant=minbase).
2. **Component Separation**:
   * Vid_Mux Container: Production application. Per-source GStreamer pipelines, Flask REST API, MJPEG stream, Web UI.
   * Vid_Mux_TEST Container: Kernel module scaffold only. Compiles v4l2loopback against host kernel, creates /dev/video200 as a readiness signal, then sleeps. No stream is written to it — vid_mux uses internal videotestsrc for all mock sources. Only launched when physical camera count < 2.
   * scanbox_dhcp Container: DHCP server for USB NCM link. Runs dnsmasq on host network, assigns 192.168.199.2 to Windows PC (/30 point-to-point).
3. **Deterministic Device Mapping**:
   * Physical cameras: /dev/video100–/dev/video103 ← /dev/v4l/by-id/*-video-index0 (assigned by rebuild_vid_mux.sh in discovery order)
   * Mock sources: slot=null in SCANBOX_SOURCES — no V4L2 device; vid_mux uses videotestsrc internally. mock_0=color bars ("MOCK 1:"), mock_1=SMPTE ("MOCK 2:").
   * Mock count: max(0, 2 − physical_count): 0 physical → 2 mocks, 1 physical → 1 mock, ≥2 physical → 0 mocks.
   * Control: Python Flask REST API on port 80
   * Output: MJPEG HTTP stream at /stream (port 80) — accessible over WiFi/LAN or USB NCM direct link
4. **Code Standards**: all discussions, code, comments, and files in English. No Spanish in any user-visible text, UI strings, or source code.
5. **Startup orchestration**: `host/scanbox.service` runs `rebuild_vid_mux.sh` at boot. `rebuild_vid_mux.sh` is the single source of truth for the full stack lifecycle: it runs `udevadm settle`, detects cameras, conditionally starts vid_mux_test, builds SCANBOX_SOURCES, and launches vid_mux. `docker compose up -d` (fallback) uses a simplified hardcoded SCANBOX_SOURCES and the shared `vid_mux` image tag.

### Dynamic Multi-Camera Support — COMPLETED ✅

**SCANBOX_SOURCES env var** — passed by `rebuild_vid_mux.sh` to the container at runtime:
```json
[
  {"id": 0, "slot": "/dev/video100", "label": "usb-046d_0809_5DD0F8C2"},
  {"id": 1, "slot": null, "label": "mock_0"}
]
```
- Physical cameras: slots `/dev/video100` → `/dev/video103` (in discovery order from `/dev/v4l/by-id/*-video-index0`)
- Mock sources: `slot: null` — no device file; vid_mux uses internal videotestsrc
- Mock count: `max(0, 2 − PHYSICAL_COUNT)` — always ≥2 total sources
- If env var is absent or invalid: fallback to auto-detect video100 + one mock source

**Camera env file** written to `/tmp/scanbox_cameras.env` (best-effort, `|| true` — may be root-owned from boot):
```
CAMERA_COUNT=1
CAMERA_0_DEVICE=/dev/v4l/by-id/usb-046d_0809_5DD0F8C2-video-index0
CAMERA_0_SLOT=/dev/video100
CAMERA_0_LABEL=usb-046d_0809_5DD0F8C2
MOCK_COUNT=1
SCANBOX_SOURCES=[...]
```

**Key architecture decisions:**
* `rebuild_vid_mux.sh` — runs `udevadm settle` before camera detection to avoid race with udev at boot; writes `/tmp/scanbox_cameras.env` with `|| true` to survive root-ownership conflicts
* `docker-compose.yml` — declares `image: vid_mux` (shared tag with rebuild_vid_mux.sh) so `docker compose up -d` always uses the last image built by the script, preventing SCANBOX_SOURCES from being overwritten
* Deploy flow for code changes: `docker compose build && ./rebuild_vid_mux.sh --skip-build`
* `Vid_Mux/switcher.py` — per-source independent pipelines; physical: MJPEG passthrough (no jpegdec); mock: videotestsrc (two patterns)
* `Vid_Mux/api.py` — display names resolved at startup; mock_0 → "Mock Camera 1", mock_1 → "Mock Camera 2"

**Note on docker-compose vs rebuild_vid_mux.sh:**
- `docker compose up -d` (boot automation via scanbox.service → rebuild_vid_mux.sh): uses dynamic SCANBOX_SOURCES built at runtime
- `./rebuild_vid_mux.sh --skip-build`: fastest re-launch after non-Dockerfile code changes (skips `docker build`)
